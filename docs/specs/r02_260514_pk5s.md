## Decisions log

### Architecture

- `Pk5sGateComputer` is a new class alongside `PKDetector`. The two coexist ‚Äî `PKDetector` continues to emit per-line PK signals into `pk_signals`; `Pk5sGateComputer` produces a directional gate that filters which of those signals survive.
- Gate output is **sign-inverted** from Pine `s5_pk_final` (Pine `+1=long` ‚Üí Python `-1`). This makes the gate output an OOB-equivalent direction array, so it plugs into `IndicatorComputer.fold_gates` alongside bny30M/p without bespoke folding logic.
- `LookaheadResampler` is a new static helper inside `IndicatorComputer`. Produces a developing-higher-TF OHLCV view at every 5s timestamp; BB and K chains compute on that view.

### Pine alignment

- `f_pk_state` math in `Pk5sGateComputer` matches Pine exactly: window covers `line[i - upper + 1 : i - lower + 1]` of length `pool_range` bars.
- Existing `PKDetector` carries a 1-bar wider window (`pool_range + 1` bars). **Intentionally left as-is** for continuity with the 30-day grind dataset. Documented in `PKDetector`'s docstring with a reference to this spec and a `# TODO: patch after next clean centroid is locked` flag.
- PM sentinels `¬±2.0` route to the neutral bucket at full weight in `f_vote` ‚Äî Pine logic verbatim.
- PM suppression is **subtractive**: `adj_long = max(0, long_pts ‚àí pm_short_wt √ó pm_suppression)`. Matches Pine `pm_suppress_str` semantic, not multiplicative dilution.
- Active-weight denominator includes neutrals (Pine `pm_option_a=false` default). The three ratios (long, short, neutral) sum to 10 when `active_w > 0`.
- **Dead zone on ratios: not added.** The `√ó 10` scaling forces ‚â•5-point ratio gap at threshold 7.5 ‚Äî already a strong condition. PM divergence is substantive directional evidence; 0.4 suppression already moderates it appropriately.

### PM suppression rationale ‚Äî captured for the codebase

> PM_LONG means "the bullish lines and the price proxy are organically aligned ‚Äî no visible divergence here." That alignment is the *opposite* of divergence; it's evidence of ongoing directional strength. The 0.4 suppression weight lets that evidence reduce opposing votes by 40% of its own weight, without itself voting directionally. Gating the verdict out when PM did the heavy lifting would contradict trusting PM evidence at all.

This paragraph goes verbatim into `Pk5sGateComputer`'s class docstring under *Design notes*.

### Trigger modes ‚Äî per-line evaluation cadence

Stored on `test_config_ext_votes.tcev_trigger_mode`:

- **`standard_pk`** ‚Äî `f_pk_state` evaluates every history-valid bar. Default. All six 5s vote contributors use this mode in the seed.
- **`roc_curl`** ‚Äî evaluates only on bars where the line's slope changed by more than `tcev_roc_threshold` degrees from the prior bar. Used by b30M / b30b in the eventual trend machine. Schema reserves the mode; no seed rows use it in this round.

#### `roc_curl` math

- **Trigger**: `|atan(line[i] ‚àí line[i-1]) ‚àí atan(line[i-1] ‚àí line[i-2])| √ó 180/œÄ > tcev_roc_threshold` (default 45¬∞)
- **Peak**: `line[i-1]` ‚Äî the spike. The bar before the curl is the local extreme by definition (it's what `ta.highest`/`ta.lowest` would find anyway).
- **Line slope**: `line[i] ‚àí line[i-1]`
- **slope_floor**: not applied. The ROC trigger itself is the noise filter.

**Open question ‚Äî DEMA anchor in `roc_curl` mode:**

Two options, implementation supports either:

- **(a)** `dema[i-1]` ‚Äî paired with the prior-bar peak. Both slopes are 1-bar deltas. Symmetric and simple, but loses the multi-bar momentum context that `standard_pk` builds in via `center = bars √ó multiplier`.
- **(b)** Pool-centred: `dema[i ‚àí center]`, same as `standard_pk`. Preserves multi-bar momentum window. Comparing a 1-bar line slope against a multi-bar DEMA slope is dimensionally awkward but only the *signs* matter for the divergence/PM decision, so the math is still valid.

My read: **(a)** is conceptually cleaner. The ROC trigger has already identified the moment as significant; the comparison is "does price agree with the line's reversal *right now*?" ‚Äî a 1-bar question. The bigger-picture DEMA context isn't relevant.

Confirm or override before implementation.

## `LookaheadResampler`

### Purpose

For each 5s bar `t`, produce the value of a higher-TF indicator line as if it were computed against the *developing* higher-TF bar containing `t`, not the last fully-closed higher-TF bar. **Pine equivalent**: `request.security(syminfo.tickerid, "30S", line, barmerge.gaps_off, barmerge.lookahead_on)`.

### Mechanism

The developing higher-TF OHLC at 5s offset `k` within its window:

- **O** = first 5s open in the window (constant across the window)
- **H** = cumulative max of 5s highs from window start through `t`
- **L** = cumulative min of 5s lows from window start through `t`
- **C** = current 5s close at `t`

A BB with `len=60, src=hl2` then sees, at each 5s bar:
- 59 fully-closed higher-TF source values (resampled series shifted by one window)
- 1 developing higher-TF source value (computed against the developing bar at `t`)

**BB**: closed-form. Maintain rolling sum and sum-of-squares over the closed 59-bar window; combine with the developing source value for the full 60-bar mean and stdev at each 5s. O(N) overall, vectorisable.

**K chain (RSI ‚Üí Stoch ‚Üí SMA)**: harder. RSI uses Wilder's smoothing ‚Äî the developing-bar RSI is one update step from the last-closed-bar state (carry `avg_g`, `avg_l`). Stoch is rolling max/min over the rolling RSI; same closed-window-plus-developing-step pattern as BB. Final SMA is trivial.

### Scope of use this round

- **b6M (TF6 = 360s)**: gets p-rev when `--p_rev on`. This is the calibration target ‚Äî PK divergence math anchors to "where the line is right now" instead of the last closed 360s bar.
- **bny30M / bny30p (TF3 = 30s)**: **do not** receive p-rev. Their job is gate-open/close at a coarse OOB resolution; precision unnecessary.
- **5s vote contributors**: not applicable (already 5s-native).

        Pine had been queried with barmerge.lookahead_on.

        """

    @staticmethod

`OptimizerRunner.run` switches dispatch on `--p_rev`:
- `on` + higher-TF line: compute via `f_bb_lookahead(base_df, ic_itf_seconds, ...)` directly. No `align_to_base` step.
- `off` or 5s-native line: behaviour unchanged from current (resample ‚Üí compute ‚Üí forward-fill).

## `ReportManager` / `OptimizerRunner` integration

`ReportManager.run` accepts `p_rev_enabled` and `pk5s_gate_enabled` (CLI plumbing in `run.py`). Behavioural changes:

- New `_load_pk5s_extensions(tc_pk)` returns active `pk_5s` tce rows with parsed JSON params
- When `pk5s_gate_enabled`: build pk_5s gate arrays via `Pk5sGateComputer.compute(...)`, append to `gate_sides` before `fold_gates`
- When `p_rev_enabled` and the calibration line's TF > 5s: pass `use_lookahead=True` to the line-compute call inside `OptimizerRunner.run`. The grid iteration reads this once and dispatches each combo accordingly.
- Both flag values recorded on the `optimizer_runs` row at run creation
